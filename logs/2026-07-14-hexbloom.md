## Idea Source
IDEAS.md (first item, now removed from the queue):
> "Hex Territory — turn-based area-control board game on a hexagonal grid for 2–4 players; claim and flip adjacent tiles."

Fleshed out into **Hexbloom** — an original take on the classic "flood / filler"
mechanic on a hex grid: pick a colour, your whole territory recolours and absorbs
every touching tile of that colour, blooming outward. Chosen because it's legible
in one sentence, deeply satisfying (cascade captures), works beautifully solo vs
AI, and is genuinely fun 2–4 P2P — with low, turn-based state that suits WebRTC.

## Game Details
- **Name / repo:** Hexbloom / ben-gy/hexbloom
- **Genre:** board
- **Core loop:** Pick a colour → territory recolours + absorbs adjacent same-colour tiles → largest territory when the board fills wins.
- **Multiplayer:** live P2P, host-authoritative full-snapshot, 2–4 players (seating derived from sorted lobby roster; host re-election; disconnect auto-play; spectators). Also solo vs 1–3 AI rivals (easy/normal/hard).
- **Stack / render:** vanilla-ts / DOM + inline SVG board + canvas particle overlay
- **Engine modules used:** net, lobby, rng, sound, storage (no loop/input — turn-based, tap-driven; rAF only for the particle/shake visual layer)

## Juice
Cascade "pop" spring on every absorbed tile, colour-scaled particle bursts,
screen shake proportional to cascade size, procedural Web Audio (bloom/turn/win/
lose + powerup on big blooms, pitch scales with capture count), tweened tile
recolours, pulsing active-player HUD chip, animated lobby/turn spinners. All
degrade under `prefers-reduced-motion` (no shake, ~no particles).

## Errors & Resolutions
- **TS build error** in copied `net.ts`: Trystero 0.21's `makeAction<T>` constrains payloads to a `DataPayload` union, which the unconstrained generic `T` didn't satisfy. Fixed by widening `makeAction` past that constraint via a local cast in the channel factory.
- **Invite-link join bug (found via the two-tab test):** `enterFriends()` called `cleanupMp()` first, which stripped the `?room=` param before `getOrCreateRoomCode()` read it — so clicking "Play with friends" from an invite link always minted a *new* room instead of joining. Fixed by moving the room-param stripping out of `cleanupMp()` into a `stripRoomParam()` that only runs on returning to the menu. Re-tested: second tab correctly joined the host's room and synced.
