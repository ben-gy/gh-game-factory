# Build Log: Ballast
**Date:** 2026-07-22
**Status:** deployed

## Idea Source
**Invented.** `IDEAS.md` was empty, so the idea came from reading the registry for gaps: 24
games and **zero action-puzzles** — no stacker, no merge, no real-time tile game — with
`puzzle` the thinnest genre at 1 of 24 (lumenlock). The stacker is the most proven "one more
go" loop there is and the factory brief explicitly blesses it, but a straight reimplementation
would be a reskin, so the run is built around one added rule.

Rejected on the way: a drawing/physics game (high build cost, weak mechanism-audit story), a
Nikoli-style logic puzzle (low risk, low ceiling), and a colour-alternating drag-a-path grid
game — that last one was cut specifically because its *input* is identical to cipher-clash's
(drag a path across a grid in a race) even though the game differs.

## Game Details
- **Name / repo:** Ballast / ben-gy/ballast
- **Genre:** puzzle
- **Core loop:** Stack falling pieces, clear rows — but the well hangs from a pivot, so the
  stack's torque makes it *list*. A listing well curves the fall (pieces slide downhill, the
  far side goes out of reach) and a level well pays double. The two pull against each other.
- **Multiplayer:** live P2P versus, 2–6, **parallel same-seed race**. No board state on the
  wire — only `st` (4Hz status), `atk` (ballast), `res` (final result). Host owns round start
  only, so host transfer is a display concern.
- **Stack / render:** vanilla-ts / canvas
- **Engine modules used:** net, rematch, turn, lobby (+qr), rng, sound, storage, mobile, identity

## Juice
The whole board is drawn inside a **rotated frame** whose angle is the list — that single
visual teaches the mechanic in the first second, and particles are emitted inside it so debris
falls along the lean. Plus: row-clear flash + shatter, screen shake scaled by rows, hit-stop
on hard drop, an eased needle on the list meter, an IN TRIM pill that flips state, a pulsing
danger rim near the ceiling, score/combo toasts, 3-2-1-GO countdown with audio, and procedural
SFX throughout. Reduced motion caps the lean at 1.2°, drops shake and particles.

## What the balance sim actually said
Built **before** any tuning. The baseline immediately overruled the plan:

1. **Careen was not a game** — 22 pieces, 0.5 rows per run. The list pinned at ±1, drift firing
   every piece at max. No drift setting rescued it; the keel had to move (18 → 40).
2. **The plan named the wrong constant.** It called `keel` "the load-bearing balance constant".
   Sweeping keel moved outcomes noisily and non-monotonically (broadside: GAP 18 / 10 / −13 /
   14 across four values). Sweeping `listThreshold` moved them monotonically, because the
   threshold decides whether drift fires *at all*. `modes.ts` now says so in the comment.
3. **Broadside's signature mechanic never fired** — the trim-blind control bot *beat* the
   trim-aware one there (181 vs 154 pieces). Fixed by keel 120 → 70 and threshold → 0.14.
4. **The drift-direction worry was the wrong question.** The plan's stated fear was that
   sliding downhill would snowball; both directions gave the same leader curve. What direction
   actually decides is whether the mechanic *exists*: uphill drift self-corrects the list, the
   firing rate collapses to 0.09/piece, and trim-awareness becomes a **liability in all three
   modes** (GAP −11 / −9 / −26). Pinned as `DRIFT_DIRECTION`.

**And one finding about the instrument, not the game.** A 5-point seat-0 deficit in trim 2P
replicated across three "independent" seed families — 44.4 / 48.6 / 44.7 — and was an artifact:
every family handed seat 0 the same rng stream, so the families were not independent in the
dimension being tested. Rotating the seat→stream schedule scatters it (47.1 / 57.9 / 50.0).
`rotateStreams` now defaults **on**, with the reason in the comment.

Two more things worth recording:
- **The control arm's GAP is negative in all three modes** (−25 / −2 / −12). With drift
  disabled, caring about trim is a pure liability. That is the proof the mechanic is
  load-bearing rather than the sailor bot merely being a better bot, and the control reports a
  **true zero** of drift events, asserted rather than assumed.
- **A 2P same-seed race with identical deterministic bots is 100% draws** — perfect symmetry.
  The very first versus run measured nothing at all (120 matches, 120 draws) until each seat
  got its own rng stream.

## Bugs found, and by which gate
Every one was invisible to everything except the gate that caught it. All mutation-verified.

| Bug | Caught by |
|---|---|
| **The I-piece could not be stood upright on the move it spawned.** A 4-tall vertical rotation needs a cell above the ceiling; the kick table only kicked *up*, so every candidate left row −1 occupied and the rotation was refused. The most important rotation in the game, silently dead. | **Playing it.** 46 unit tests were green. |
| **IN TRIM green vs LISTING amber at 1.14:1** — the game's most important readout, told apart by hue alone | `contrast.test.ts` |
| **The list-meter needle moved half a pixel.** `translateX(%)` resolves against the element's own 4px width, so the meter was decoration. | Reading a screenshot |
| **The join QR was yanked off screen ~1s after tapping it.** `showLobby()` rebuilt the engine lobby on every roster/vote change, and a fresh lobby starts with `qrOpen` false — the engine's own guard against exactly this, defeated from outside it. | Looking at the lobby |
| **A late net-handler repaint painted the lobby over the results screen.** `rounds.finish()` reopens voting the moment a round ends, so nobody ever saw anyone's breakdown. | Two-peer smoke test |
| **A stale lobby handle left the host in an empty lobby through its own rematch** — introduced *by* the QR fix, when a stale handle met a container emptied by a results repaint. | Two-peer smoke test |

The last two are the memory's "net-handler repaints must not navigate" failure mode, in both
of its documented forms.

## Errors & Resolutions
- **The harness's browser pane is permanently `document.hidden`, so rAF never fires.** The
  game's loop is rAF-driven, which meant no verification was possible through it. Resolved
  properly rather than worked around: the frame body was extracted into a shared `tick()` that
  both the rAF loop and an external step hook call, so a frame driven from the console is the
  *same code path* — and a visibility-aware `setInterval` backstop was added so a real browser
  that starves rAF while the page is on screen cannot silently freeze the round.
- **A file-copy clobbered `tests/helpers/sim.ts`** while pulling the standard contract tests
  across from unstrung. Rewritten from context and re-verified against the same numbers.
- **The contrast pixel probe could not reach the renderer's metrics** through the runner, and
  synthetic pointer events do not reliably reach a canvas. Resolved with the throwaway
  `harness.html` the brief recommends — built as a temporary second Vite entry so it served
  from the already-approved preview origin — then deleted.
